对象：学生端“学习内容记录”方式 A

建议接口与任务：
- `upload_work_effort_api(user_id, syllabus_id, files)`：方式 A 专用上传接口。
- `tasks.learning_task.upload_work_effort(...)`：不要复用 `upload_knowledge_source()` 作为业务入口；只能复用底层落盘 / file 表登记能力。
- `work_effort_status_api(user_id)`：返回 `{user_id}_learn_cache.json` 中仍在处理的记录数，作为正在处理的学习记录数量。

上传：
- dropbox 与接口都限制最多 10 个文件。
- 每个文件由 `upload_work_effort` 开一个独立子任务维护。
- 文件落盘并写入 mysql `file` 表。
- 子任务创建后 API 尽早返回；后端继续维护最多 10 个子任务。

解析：
- 每个子任务提交一个 `end_stage=pdf_to_md` 的解析 job。
- 记录 `job_id` 到 `{user_id}_learn_cache.json`，文件内直接维护列表字段。
- 当 job `completed` 后，读取解析得到的 markdown 结果，并进入学习评估阶段。

学习评估：
- 将 markdown 结果与教学大纲 raw json 一并提交给 LLM。
- LLM 输出必须能直接驱动 `update_personal_syllabus`。
- 采用直接推进 `competance_progress` 的形式更新，`week_index=-1` 表示由 LLM 输出决定要更新哪些周次。
- 单个文件最多允许带来 `+3` 的 progress。
- 单个子任务完成后，从 `{user_id}_learn_cache.json` 清除对应 `job_id` 记录。

补充调整：
- `update_personal_syllabus` 在使用 `competance_progress` 调整时，应从“覆盖原有值”改为“在对应 week_index 上累加 n”。
- 累加后仍由 `update_personal_syllabus` 内部处理达到阈值后的晋级 / 降级逻辑。

总体难度：中到高

难点：
- 子任务生命周期管理与 `{user_id}_learn_cache.json` 一致性。
- pdf_to_md job 状态轮询 / 回调后的继续处理。
- LLM 输出 schema 必须稳定，否则会误更新个人学习进度。
- `week_index=-1` 的批量更新语义要非常明确，避免绕开前面已确定的“方式 B 必须指定具体周次”的约束。
