2026年4月11日 vibecode 驱动备忘

用途：
记录本轮 API 对齐里原先存在重大歧义、现已完成收口的部分，便于后续继续联调时直接按当前实现推进。

1. `generate_material_draft_api()` 的前端调用已去掉 `materialId`
前端不再预造虚拟 material。
生成草稿后，以后端返回的真实 `material_id` 为准，再继续拉取 `material_draft` / `material_status`，并进入后续编辑链。

2. 前端真实请求不再固定传 `user_id=7`
当前实现改为从 `localStorage` 读取登录态。
若不存在本地登录态，教师端 / 学生端会自动跳回 `/login`。
登录 / 注册成功后，会将 `user_id`、`user_name`、`email` 写入 `localStorage`，供后续真实 API 请求复用。

3. 教师端教学材料上传后的新卡片，已改为先拿 `file_id`，再调 `get_file_detail_info`
后端已补齐：
`tasks.file_task.get_file_detail_info(file_id)`
`POST /api/file/detail`
前端当前链路为：
`upload_knowledge_source()` -> `get_file_detail_info()` -> `create_job_api()`
因此上传后的新卡片标题不再取本地 `file.name`，而是以服务端 file 详情为准。

4. 文件展示对齐约束
凡是接口本身只能先返回 `file_id`、而页面又需要展示 `filename/path` 等内容的场景，应统一补一次 `get_file_detail_info(file_id)`，不要再以前端本地文件对象兜底。
当前教师端教学材料上传已按此约束收口；`list_graph_files_api()` / `list_syllabus_files_api()` 这类本身已返回完整文件简表的接口，不需要额外逐条补查。
